## Your 5 highest-value answers (bold opener verbatim, then the follow-through)

**1. "How many agents are on Agentverse — show me a message between two of them?"**
> **"Three — Watcher, Librarian and Operator are registered Agentverse agents passing uAgents messages; every ASI:One reply carries the hop trail so you can verify it yourself."**
The Assessor and Auditor run in-process by design — the deterministic policy and audit components — because we keep permission decisions out of network hops. **Conditional:** if only two registered, say **"Two on Agentverse — the gateway and the executor; retrieval runs in-process, deliberately, to keep ACL decisions off network hops."** Never say three if two are registered.

**2. "Close your console — resolve an incident end-to-end inside ASI:One."**
> **"One conversation: you report the failure, the agent replies with triage, the retrieved documented fix, risk class and rollback plan as one message; you type 'approve'; it executes, the real Jira ticket transitions and closes, and it returns the audit-trail links."**
The Standing Approval repeat runs in the same session in ~15 seconds. The public shared-chat URL of exactly this flow is in the submission — no custom frontend needed.

**3. "In an ASI:One chat the approver is an anonymous session — who's the authorising principal when I type 'approve'?"**
> **"The Chat Protocol sender address — recorded verbatim as the authorising principal in the audit record."**
Requester (agent) ≠ approver (chat principal), immutably logged; in an enterprise deployment that address maps to SSO identity. A distinct, logged principal who is not the agent is fully present in the chat path.

**4. "The session drops between propose and approve — does the pending execution leak?"**
> **"It expires, closed."**
A proposed plan is a pending approval object with a TTL; no approval message, no execution — ever. On reconnect the agent re-presents the plan rather than resuming silently. A dropped session can only ever produce a non-action — the correct failure direction for an execution gate.

**5. "What does Fetch.ai actually do that FastAPI wasn't already doing?"**
> **"It's the agent transport and the front door — discovery, identity, and the conversational approval surface."**
The agents communicate over uAgents messaging; the Watcher is discoverable from ASI:One with no custom frontend; and the chat sender address is load-bearing — it's the authorising principal in our audit log. Agentverse is where the human-in-the-loop actually lives.
