Security posture

A small surface, and everything on it named.

This is the seeded posture summary — the parts a security reviewer asks for first. It is real content, not a template: the dependency list, the no-secrets proof, and the single-container argument are all drawn from the shipped repository. The full posture page (threat model, data-flow diagram, sub-processor list) is the next milestone.

1 · Dependency surface

The runtime is ten direct dependencies, each installed from a prebuilt wheel — no compiler in the image, no native build step (numpy and pandas carry compiled extensions; their wheels ship those prebuilt). There is no closed or proprietary model SDK anywhere in the tree: every inference call goes over the Venice OpenAI-compatible HTTP API through httpx, and the only file permitted to name a model is precedent/models.py.

  • fastapi · uvicorn — the sim + console server
  • pydantic — typed message contracts across every boundary
  • httpx — the only outbound HTTP client (Venice / Jira)
  • python-dotenv — local .env loading
  • sse-starlette — the console's server-sent event stream
  • jinja2 — server-side page rendering
  • pandas · numpy — deterministic bench + bitmap ops
  • pyyaml — the deterministic policy pack

The Fetch.ai rails, the browser end-to-end harness and the test tooling are all optional extras — the product container installs none of them.

2 · No secrets, and it is proven

  • Nothing committed. .env and .env.* are gitignored; only .env.example — placeholder names, no values — is tracked.
  • History-scanned. make secrets-scan runs gitleaks across the full git history on every push; the honesty gate fails closed on any hit.
  • No key in the image. No VENICE_API_KEY is baked into the container. With no key present, every model call falls back to a canned rationale — so the hosted demo makes zero outbound model calls and burns no credit.
  • No model in the decision. Permission and risk are decided by the deterministic policy engine and a human identity — never by a model — so a leaked inference key could never widen access.

3 · One slim container

The whole product ships as a single python:3.13-slim image. It runs the MediaCo simulator bound to loopback only and the console on the public port, in one process group — there is no database service, no message broker, no sidecar to patch. Runtime dependencies are installed in their own layer before the source copy, and the test tree is excluded from the build context, so the installed surface is those ten direct dependencies — with their pinned transitive dependencies — plus the application code.

Fail-closed is the default everywhere: when access-list freshness is uncertain, restricted memory is denied rather than served, and a stale cache may only narrow access, never widen it.
