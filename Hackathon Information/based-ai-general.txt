# BasedAI at UK AI Agent Hack EP.5

**Who we are**
BasedAI is the commercialization layer for open source AI — we turn open-weight models into production-ready products and enterprise APIs. Our stack: BasedAPIs (production inference) and Hirebase (run what you build without hiring a team around it). We sponsor EP.5, and we run a track plus two workshops this week.

## Track — Enterprise Memory Governance at Scale
Agents do better work with a shared organizational memory. The challenge: give many agents a persistent shared memory without leaking across boundaries (orgs, teams, projects) at scale.

Build a permission-aware memory layer that:
• syncs with source ACLs under concurrent updates
• enforces access at the retrieval layer with a deterministic permission path (no LLM in the decision)
• produces audit logs meeting regulatory standards
• holds sub-200ms permission checks at P99
• governs derived memory by lineage — revoke a source and it propagates to everything derived from it

**Requirement:** build with open-weight models only.

# How to Submit

All submissions for the BasedAI track (Enterprise Memory Governance at Scale) are made through the BasedAI hackathons repository:
https://github.com/BasedAICo/hackathons

We use a pull-request-per-team model.

## Steps
1. Fork the repository.
2. Copy the template folder UK-AI-Agent-EP5/submissions/_TEMPLATE/ to UK-AI-Agent-EP5/submissions/<your-team-name>/.
3. Add your project code to that folder and fill in its README.md — it is the first thing judges read, so keep it clear. It should cover:
   - Project name, team, and members (GitHub handles)
   - What it does — the problem and your approach
   - A demo video link (plus a live demo and screenshots if you have them)
   - How to run it (prerequisites, install, configure, run)
   - How it works — architecture, agent framework, models, and APIs used
   - The open-weight models you used (required — no closed, proprietary models in the loop)
4. Open a Pull Request into main. Your PR should only touch files inside your own team folder.

## Rules and reminders
- Event-specific rules (bounty, deadline, judging) are in UK-AI-Agent-EP5/README.md; the full contribution guide is in CONTRIBUTING.md.
- This is a public repository. Never commit secrets — no API keys, .env files, or crypto wallet private keys or seed phrases. Use .env.example to document the variables you need and keep real values in your own local .env.
- Your submission must run on open-weight models only.
- Submissions close at Demo Day on 4 July — open your PR before judging begins.

Direct submission link: https://github.com/BasedAICo/hackathons/tree/main/UK-AI-Agent-EP5

[02/07/2026 07:30] santanasupplied
Yes, I think A and B collapse into the same thing only in the simple case where permissions are a single linear sensitivity ladder, e.g. public → internal → confidential, and everyone with the higher clearance can access everything below it.

Where I think they diverge is when permissions are non-hierarchical or source-specific.

Example: a Phase II memo is derived from a SAR table, a toxicity report, and an adverse-event memo. The SAR table may be R&D-only, the adverse-event memo may be Regulatory-only, and a CRO may have access to the CRO assay report but not either of those. If we just say the memo inherits the “strictest” label, we may lose the fact that access depends on multiple separate source constraints, not one global level.

So my current thinking is:

A = safest default semantics: to read a derived artifact, the user must satisfy the source lineage constraints.

B = useful implementation optimization: precompute an “effective policy” or “max sensitivity” from the parents so retrieval checks stay fast.

C = only for deliberately redacted/declassified derivatives, where the system creates a new governed artifact with an attestation saying which sources were included/excluded.

Would that fit how you’re thinking about Hirebase, i.e. A as the semantic rule, B as a cached policy representation, and C only for explicit redacted artifacts?


[02/07/2026 07:51] c0b4lt_27
Yes. I believe that would cover the various permutations. The question then would be who decides which to use when. Would the user decide? Would the system decide? Or would the system recommend and the user approve? Or another option.


[02/07/2026 08:21] santanasupplied
I don’t think the end user should decide the mode at query time, because that would turn the access model into a UI/prompt-level choice.

My current thinking is:

1. The organisation/workspace admin or data owner defines the policy template.
2. The system applies the safest default automatically.
3. A normal derived artifact uses A as the semantic rule: access must satisfy the source lineage constraints.
4. B is the optimised representation of A: the system compiles/caches the effective policy or strictest constraint so checks stay fast.
5. C is only for deliberate redacted/declassified derivatives. The system can recommend that path, but a user/service with an explicit redact/declassify capability should approve it. The result gets a redaction attestation, lineage record, source hashes, and audit event.

So I’d frame it as:

- User/agent can request a derived artifact.
- System chooses the safe inherited policy by default.
- Admin/data owner policy decides whether redaction/declassification is allowed.
- Authorised human or service approves weaker derivatives.
- Runtime enforcement remains deterministic.

Would that make sense for Hirebase: policy template chosen at workspace/admin level, system-enforced at artifact read/retrieval time, and exceptions handled through explicit governed redaction rather than ad hoc user choice?
